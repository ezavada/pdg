Detailed Description

animated, moving image that is automatically moved, animated, and drawn

Note
API Stability: 2 - Unstable. The API is in the process of settling, but has not yet had sufficient real-world testing to be considered stable. Backwards-compatibility will be maintained if reasonable.

Events

Things that happen to a sprite that might require intervention, such as collisions, completed animations, going offscreen, etc... will generate events. Events generated by sprites first go to any handlers attached directly to the Sprite, then propagate up to the SpriteLayer and then the EventManager until they are handled.

Warning
translating events to JavaScript and running them through numerous handlers is expensive. Try to put your handlers as early in the propagation chain as possible.

Helpers

You can add helper functions or classes to do specialized animation behavior, prefilter collisions, and add visual effects during drawing.

See also
setAnimationHelper()
setDrawHelper()
setPostDrawHelper()

◆ activateSubEntity()

activateSubEntity ( string entityName,
string animationName = "idle" )

activates a sub-entity with the specified name and animation

This function activates a sub-entity within the Spriter animation system. Sub-entities are additional animated components that can be controlled independently of the main Spriter animation.

Parameters
entityNameThe name of the sub-entity to activate
animationNameThe name of the animation to play on the sub-entity

Sub-entities are typically used in complex character animations where different parts of the character (like arms, legs, or accessories) need to be animated independently. Each sub-entity can have its own set of animations and can be controlled separately from the main sprite.

When a sub-entity is activated, it will start playing the specified animation. The sub-entity will continue playing until it's deactivated or another animation is specified.

Warning
If the entityName or animationName is invalid, the function will fail silently
Note
Sub-entities are only available for sprites that support them (typically Spriter animations)
The sub-entity will be positioned relative to the main sprite's position and rotation
Returns
true if the sub-entity was successfully activated, false otherwise
See also
Sprite::deactivateSubEntity()
Sprite::isSubEntityActive()

◆ blendToAnimation() [1/2]

blendToAnimation ( int animationId,
number blendTime )

Smoothly blend from current Spriter animation to a new Spriter animation by ID over the specified time.

This function smoothly transitions from the current animation to a new animation by interpolating between the two over the specified blend time. The blend creates a seamless transition between animations.

Parameters
animationIdThe ID of the target animation to blend to
blendTimeThe duration of the blend transition in seconds

The blend operation interpolates between the current animation frame and the corresponding frame in the target animation. This creates a smooth transition that maintains the sprite's visual continuity.

During the blend, the sprite will emit animation events and can be monitored for blend completion. The blend time determines how quickly the transition occurs.

Warning
If the target animation ID is invalid, the blend will fail silently
This only works for blending between Spriter animations
Note
The blend time must be positive (greater than 0)
If no animation is currently playing, the new animation will start immediately
Returns
true if the blend was successfully initiated, false otherwise

◆ enableCollisions()

Sprite enableCollisions ( int collisionType = collide_AlphaChannel)

start checking for collisions with this sprite

Collision Types are:

  • collide_None – actually calls disableCollisions()
  • collide_Point – just collide with the centerpoint of the sprite. Very fast.
  • collide_BoundingBox – collide with the boundaries of the sprite (rotated if necessary). Reasonably fast.
  • collide_CollisionRadius – collide with all points within a given radius of the sprite (best guess at radius based on sprite height and width is used unless setCollisionRadius() is called. Very fast.
  • collide_AlphaChannel – per-pixel collision based on the alpha channel of the sprite's current frame. This is the most accurate, but also the slowest. Avoid it if another collision check gives acceptable results, especially with fast-moving items where the difference between a hit and near miss is not obviously visible to the player.
Warning
you must have called Image.retainAlpha() on the sprite's image(s) in order to use collide_AlphaChannel

References pdg::collide_AlphaChannel.

◆ gear()

CpConstraint gear ( Sprite otherSprite,
number gearRatio,
number initialAngle = 0,
number breakingForce = 0 )

join two sprites as if they were geared together (Chipmunk Physics Only)

Whenever this sprite spins, it will force the other to spin the same amount (adjusted by the gear ratio).

If a breaking force is given, then the joint will break, generating a SpriteBreakEvent, if greater than that amount of rotational force (torque) is applied to the joint.

Returns
the cpConstraint for the Joint in case you want to do anything special with it
Note
this only affects rotation, it does not join them in any other way, so they are still free to move around independently otherwise.

◆ setCollisionHelper()

setCollisionHelper ( ISpriteCollideHelper helper)

set a helper to filter out some collisions for this sprite

Sets up a handler to do special case collision logic between two sprites. For example, you might want bullet sprites to collide with enemy sprites but not with one another. If a collide helper is installed, then on initial contact between two sprites the collide helper for the moving sprite will be called, and your helper can decide if the collision should happen or not.

Parameters
helperan ISpriteCollideHelper instance that will filter some collisions.
Note
You should use other techniques wherever possible to eliminate unnecessary collision helper callbacks, since these callbacks are relatively expensive. You should turn off collisions for sprites that never collide with anything; put sets of sprites that collide only with each other in separate layers; and use collision groups to set up groups of sprites that collide with other things but not each other.
var myHelper = new pdg.ISpriteCollideHelper(function(sprite, withSprite) {
console.log("in my sprite collide helper for " + sprite );
if (withSprite.id == 100) {
return true; // only collide with the player sprite (id 100)
}
return false; // don't collide with anything else
});
mySprite.setCollisionHelper(myHelper);

◆ useCollisionMask()

useCollisionMask ( Image frameImage,
Image maskImage )

sets collision masks for precise collision detection using image data

This function enables pixel-perfect collision detection for the sprite by using image data to define collision areas. This provides more accurate collision detection than simple rectangular bounds.

Parameters
frameImageThe image containing the frame data to use for collision detection
maskImageThe image containing the collision mask (white pixels = collision, black pixels = no collision)

The collision mask system uses the alpha channel or brightness of the maskImage to determine collision areas. White pixels (or pixels with alpha > 0) are considered solid collision areas, while black pixels (or pixels with alpha = 0) are considered empty space.

This allows for precise collision detection that matches the visual appearance of the sprite, rather than using simple rectangular bounds.

Warning
Collision mask detection is more computationally expensive than rectangular collision detection
Note
The maskImage should be the same size as the frameImage
Only the alpha channel or brightness of the maskImage is used for collision detection
